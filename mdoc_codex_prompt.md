Ты — автономный кодовый агент, запускаемый в корне репозитория. Твоя задача — ПРОЙТИ ВСЮ ДЕРЕВЯННУЮ СТРУКТУРУ ПРОЕКТА РЕКУРСИВНО, сгенерировать документацию по каждому модулю и собрать агрегаты вверх по иерархии до корня. Документация должна быть понятна LLM и человеку и СТРОГО соответствует эталонным примерам из репозитория:
- Пример ЛИСТОВОГО модуля: `leaves_readme_example.md`
- Пример АГРЕГИРОВАННОГО файла: `aggregated_readme_example.md`
- Общее описание: `README.md`

## 0) Общие принципы
- Язык документации: Русский.
- Идемпотентность: многократный запуск не должен ломать структуру; перезаписывай только `README.generated.md` и `README.aggregated.md`.
- Сохраняй ручные файлы пользователя: НИКОГДА не перезаписывай `README.md`; если он есть — учитывай, но не меняй.
- Выводи кратко и структурировано, с приоритетом на **LLM-дружелюбный формат**: JSON фронт-маттер + стабильные заголовки, маркированные списки, явные интерфейсы и зависимости, чёткие ссылки.
- Ссылки делай **относительными**.
- Если данные недоступны/сомнительны — помечай `unknown` или `[]`, но не выдумывай.

## 1) Обнаружение модулей
Считай модулем любую директорию, в которой есть хотя бы один исходный файл кода (например: `.py`, `.ts`, `.tsx`, `.js`, `.go`, `.rs`, `.java`, `.cs`, `.cpp`, `.h`, `.hpp`, `.rb`, `.php`) ИЛИ поддиректории с кодом.
Игнорируй директории и файлы по маскам:
- `.*` скрытые (кроме конфигураций в корне проекта, если нужны),
- `node_modules`, `.venv`, `venv`, `.mypy_cache`, `.pytest_cache`, `dist`, `build`, `target`, `.next`, `.cache`,
- артефакты CI/CD и бинарники (`*.o`, `*.a`, `*.so`, `*.dll`, `*.dylib`, `*.exe`),
- большие данные и медиаконтент (`*.png`, `*.jpg`, `*.pdf`, `*.csv`, `*.parquet`, и т.п.).

## 2) Для КАЖДОГО модуля создай/обнови `README.generated.md`
Структура должна **строго** соответствовать листовому примеру (`leaves_readme_example.md`) и включать фронт-маттер JSON и разделы ниже (с такими же заголовками и порядком). Если это НЕ лист (т.е. есть дочерние модули), всё равно создай файл по этой же схеме, но поля `public_api`, `inputs/outputs` и т.п. заполни из реального кода текущего уровня; про дочерние модули — только в агрегате (см. п.3).

**Формат `README.generated.md`:**

(строка с тремя дефисами не ставится; начинаем сразу с JSON-объекта)
```json
{
  "module": "<имя_модуля = имя директории>",
  "owner": "<владелец_если_известен_иначе 'unknown'>",
  "audience": "developer",
  "purpose": "<1–3 коротких предложения о назначении>",
  "public_api": ["<функции/классы/эндпойнты/CLI-команды>"],
  "inputs": ["<внешние входы: события, запросы, очереди, файлы>"],
  "outputs": ["<внешние выходы: ответы, события, записи в БД>"],
  "config": {
    "env": ["<переменные окружения>"],
    "files": ["<конфигурационные файлы>"],
    "flags": ["<CLI-флаги/feature flags>"]
  },
  "dependencies": {
    "internal": ["<внутренние модули проекта>"],
    "external": ["<библиотеки/сервисы/БД/очереди>"]
  },
  "contracts": {
    "preconditions": ["<что должно быть истинно до вызова>"],
    "postconditions": ["<что гарантируется после>"],
    "invariants": ["<что должно оставаться неизменным>"]
  },
  "quality": {
    "tests": "<есть/нет/файлы тестов>",
    "coverage": "<если известно или 'unknown'>",
    "linters": ["<flake8/eslint/etc>"]
  }
}
```

### 1. Обзор
Краткое описание назначения модуля и его места в системе (1–2 абзаца).

### 2. Контекст и границы
- Что входит/не входит в модуль.
- Основные сценарии использования.

### 3. Структура каталога
Приведи дерево файлов (до глубины 2–3), скрывая игнорируемые пути.

### 4. Зависимости
- Внутренние: `project/submodule/...`
- Внешние: пакеты/сервисы (версионирование, если известно).

### 5. Публичный API
Опиши сигнатуры и краткое поведение:
- Функции/классы/эндпойнты/CLI: по одному пункту на элемент.
- Для веб-эндпойнтов: метод, путь, краткое описание, вход/выход, коды ошибок.

### 6. Запуск и использование
- Команды запуска/пример кода.
- Переменные окружения/конфиги, без раскрытия секретов.

### 7. Поток данных (Mermaid)
```mermaid
graph TD
    <узлы и стрелки для основных сценариев этого модуля>
```

### 8. Инварианты и ограничения
Критичные предположения, лимиты, производительность (если известно).

### 9. Тестирование
Как запускать тесты, где лежат, что покрывают (если известно).

### 10. Известные проблемы и TODO
Краткий список, если найдены в коде/issue-тегов.

### 11. Связанные модули
Ссылки на соседние/используемые модули через относительные пути.

**Примечание:** Всю фактическую информацию извлекай из кода, конфигов, pyproject.toml/package.json/go.mod, Docker/CI, комментариев, тестов и маршрутизаторов (FastAPI/Express/etc.). Не догадывайся — указывай unknown.

## 3) Для КАЖДОГО родителя создай/обнови README.aggregated.md
В каждой директории, где есть подмодули, создай агрегат по образцу aggregated_readme_example.md.

**Формат README.aggregated.md:**
- Заголовок и краткое описание подсистемы (директории).
- Раздел «Обзор модулей» с подпунктом на каждого прямого ребёнка (порядок: по имени).
  Для каждого:
  - Назначение (1–2 предложения),
  - Публичные интерфейсы (короткий список),
  - Ключевые зависимости (внутренние/внешние),
  - Конфигурация (env/файлы),
  - Тесты (есть/файл/unknown),
  - Ссылка: [Подробнее о модуле "<name>"](./<child>/README.generated.md).
- Раздел «Общие зависимости» подсистемы (если есть).
- Раздел «Как использовать» (как стыковать модули между собой).
- Раздел «Развертывание и запуск» (если у уровня есть свои инструкции).
- Опционально: сводная диаграмма связей модулей:

```mermaid
graph TD
    subgraph <текущая_директория>
      <child1> --> <child2>
    end
```

## 4) Сборка до корня
- Выполни шаг 2 в каждой директории-модуле.
- Затем в КАЖДОЙ директории, где есть хотя бы один дочерний модуль, выполни шаг 3.
- Поднимаясь к корню, обновляй агрегаты на каждом уровне, пока не будет создан README.aggregated.md в корне проекта (если у корня есть дочерние модули). Корневой README.md НЕ трогай.

## 5) Критерии соответствия примерам
- Заголовки, структура, тон и формат должны повторять предоставленные эталоны.
- В README.generated.md обязателен JSON-фронт-маттер ровно с указанными ключами (можешь добавлять поля, только если нашёл точные факты).
- В агрегатах — лаконичные резюме по детям + ссылка на их README.generated.md.
- Диаграммы Mermaid создавай только если можешь вывести адекватные узлы/стрелки, иначе ставь минимальный каркас с пометками unknown.

## 6) Технические детали извлечения данных
- Языки: Python, JS/TS, Go, Rust, Java, C#, C/C++, Ruby, PHP. Ищи:
  - Точки входа (main, __main__, CLI),
  - Веб-маршруты (FastAPI/Flask/Django/Express/Nest/Fiber/Gin/Spring),
  - Объекты ORM/миграции, схемы pydantic/JSON Schema/OpenAPI,
  - Конфиги (pyproject.toml, setup.cfg, requirements*.txt, package.json, tsconfig.json, go.mod, Cargo.toml, pom.xml, Dockerfile, CI YAML),
  - Тесты (tests/, *_test.*, test_*.py),
  - Импорты/зависимости для графа модулей.
- Формируй относительные пути в ссылках, валидируй, что файлы существуют.

## 7) Политика перезаписи и безопасность
- Перезаписывай: README.generated.md, README.aggregated.md.
- Никогда не перезаписывай: README.md, .env, секреты, миграции.
- Не раскрывай значения секретов; только перечисляй имена переменных.

## 8) Итоговый отчёт в консоль (для логов ночного запуска)
По завершении выведи сводку:
- Кол-во модулей сгенерировано,
- Кол-во агрегатов собрано,
- Перечень директорий, где были изменения,
- Список предупреждений (unknown/пропуски).
